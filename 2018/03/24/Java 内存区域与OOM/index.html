Java 内存区域与OOM

运行时数据区域

程序计数器(Program Counter Register)

当前线程所执行的字节码的行号指示器，
即存放当前线程的下一个指令
唯一一个不会出现 OOM 的位置
  • 存在位置 ：线程私有
  • 不会出现OOM

虚拟机栈(VM Stack)

每个方法在执行时都会创建一个栈帧，用于存放局部变量，操作数栈，动态链接，方法出口等信息
每个方法调用直至执行完成的过程，就对应着一个栈帧进栈和出栈的过程
局部变量可以存放基本数据类型和对象的引用（不是对象本身）
64 位的 double 和 long 会占用 2 个局部变量的空间，其余 1 个。他们都在编译期完成分配
  • 存在位置 ：线程私有

  • 引发异常

    1. StackOverflowError 

      如果线程请求的栈的深度大于虚拟机所允许的深度，出现此异常（常见递归）

    2. OOM

      当可动态扩展的内存超过虚拟机内存时出现此异常

本地方法栈(Native Method Stack)

与VM Stack 类似，区别是 VM Stack是执行Java 方法，而Native Method Stack 是执行本地方法的
  • 存在位置 ： 线程私有
  • 引发异常与 VM Stack 一致

堆(Heap)

Java 虚拟机所管理的内存最大的一块
基本所有的对象都在这里分配内存
GC 管理的主要区域
Java 堆可以处于物理上不连续的空间，只要逻辑上连续即可
  • 存在位置 ： 所有线程共享
  • 引发异常 OOM 当堆无法再扩展并且没有内存完成实例分配时出现OOM

方法区(Method Area)

存储已经被虚拟机加载的类信息，常量，静态变量，即时编译器编译后的代码等数据。
方法区还存在一个 运行时常量池，用于存放编译产生的各种字面量和符号引用
运行时常量池一个重要的特征是具备动态性，Java语言并不要求常量只有编译器才能产生，运行时产生的也可放入常量池。（String类的 intern() 方法）
  • 存在位置 ： 所有线程共享
  • 引发 OOM 异常，当方法区 或者 常量池无法满足内存分配时

对象创建过程

  1. 遇到 new 时，首先检查这个指令的参数能否在常量池中定位到一个类的符号引用，并判断它所代表的类是否被加载过，没有的话先类加载
  2. 类加载通过后，开始分配堆内存
  3. 将分配的堆内存初始化零值
  4. 给对象进行必要设置，添加头信息（对象哈希，GC分代年龄等）
  5. 以上工作完成后，从JVM 的角度，一个对象就产生了，但从Java 代码的角度看，对象创建才开始。开始执行 init 方法。执行完毕后，一个对象就创建了

对象在内存中的布局

  • 对象头：

    1. 自身的运行时数据，如HashCode，GC分代年龄，锁状态标志，线程持有的锁等
    2. 类型指针，即对象指向它的类原数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例

  • 实例数据

    代码中定义的那些字段

  • 对齐填充

    无实际意义，仅仅是为了保证数据是 8 的倍数

对象的访问定位

当方法入栈后，JVM 要根据栈上 refrence 引用去找到堆中的对象，怎么定位对象在堆中的位置，有以下两种方式

句柄访问

堆中会开辟一段空间来作为句柄池，refrence 存的就是句柄的位置，
句柄中存放着对象的真实地址和对象的数据类型，这样通过句柄就能找到对象的真实位置和类型了

指针访问

这种方式 refrence 就是直接存放的对象在堆中位置
对象的头信息中存放了数据类型信息

比较以上两种方式：句柄访问在垃圾回收时，只用改变句柄中的信息即可。指针访问因为直接访问对象地址所以速度很快。这两种方式都有JVM 厂商用到。
